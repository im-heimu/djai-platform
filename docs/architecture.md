# Черновик архитектуры

Это стартовый черновик архитектуры DJAI Platform. Здесь описана целевая верхнеуровневая схема, а не уже готовая система.

English companion: [architecture.en.md](architecture.en.md)

## Цели

- Сделать on-prem AI платформу для инфраструктуры, которую контролирует сама организация.
- Разделить ключевые зоны: доступ, model access, retrieval, orchestration и администрирование.
- Упростить интеграцию с внутренними identity, data и operational системами.
- Сохранить модульность, чтобы не зафиксировать проект на одном вендоре или одном deployment-сценарии.

## Принципы

- **On-prem first:** платформа должна исходить из того, что инфраструктура и границы безопасности находятся у заказчика.
- **Ясные границы:** основные части системы должны быть отделены друг от друга.
- **Простая эксплуатация:** поведение системы должно быть понятным оператору без скрытых control plane.
- **Security by default:** auth, authorization, audit и работа с secret'ами должны быть частью базового дизайна.
- **Практичная расширяемость:** новые интеграции лучше добавлять через понятные интерфейсы, а не через жёсткую связность.

## Основные компоненты

### API/gateway

Единая точка входа для API. Здесь же могут жить маршрутизация, базовые ограничения запросов и аудит входящего трафика.

### Auth/integration layer

Связка с корпоративным identity и внутренними сервисами. Типичные задачи: SSO, service-to-service auth, mapping ролей и контроль интеграций.

### Model access layer

Единый слой доступа к model backend'ам. Он должен скрывать различия между локальными model, приватными endpoint'ами и разрешёнными внешними провайдерами.

### Knowledge/RAG layer

Слой для ingest, индексирования, retrieval и grounding. Важны границы доступа, трассировка источников и контроль над данными.

### Agents/tools layer

Слой orchestration для agent-сценариев и tool execution. Здесь нужны явная регистрация tools и понятные ограничения на действия.

### Admin/UI layer

Операторский UI для настройки системы, управления интеграциями, политиками и базовой видимостью по работе платформы.

### Deployment layer

Набор deployment-артефактов, шаблонов окружения и инфраструктурных договорённостей для on-prem установки.

## Обзор потока данных

Типовой путь запроса выглядит так:

1. Пользователь или внутренний клиент проходит auth на входе в платформу.
2. Gateway проверяет доступ и направляет запрос в нужный сервисный путь.
3. Дальше запрос может пойти напрямую в model access layer или через knowledge/RAG и agents/tools.
4. Если нужен retrieval, knowledge layer возвращает разрешённый контекст.
5. Если нужен tool execution, agents/tools layer вызывает разрешённые интеграции под заданной политикой.
6. Ответ, audit-события и telemetry уходят в систему наблюдаемости.

## Предположения для on-prem deployment

- Платформа разворачивается в инфраструктуре заказчика, а не в vendor-managed SaaS.
- Нужно учитывать среды с ограниченным outbound-доступом и по возможности изолированные контуры.
- Понадобится интеграция с внутренними identity, certificate, secret и logging системами.
- Хранение данных, retention и backup должны оставаться под контролем оператора.
- Архитектура должна подходить и для небольшой стартовой установки, и для более сегментированного контура.

## Замечания по безопасности

- Лучше опираться на корпоративные identity-механизмы, а не на отдельное хранилище пользователей проекта.
- Authorization должен быть явным для API, knowledge-доступа, tool execution и административных действий.
- Secret'ы не должны жить в репозитории; нужен внешний secret store или контролируемая инъекция окружения.
- Audit logging должен покрывать чувствительные операции и доступ к данным.
- Границы хранения и перемещения данных должны задаваться оператором.

## Замечания по observability

- Нужны структурированные логи для gateway, orchestration, retrieval и административных действий.
- Нужны метрики по состоянию сервисов, задержкам, throughput и зависимостям.
- Желательна трассировка сквозного пути: gateway -> retrieval -> model -> tools.
- Audit trail лучше держать отдельно от обычной технической telemetry.

## Замечания по расширяемости

- Подключение model provider'ов лучше строить через стабильный абстрактный слой.
- Knowledge-коннекторы стоит добавлять через явные интерфейсы ingest и retrieval.
- Tools должны подключаться через регистрацию и понятные permission boundary.
- Deployment-артефакты должны оставаться простыми для чтения и адаптации.

Главный риск на раннем этапе — слишком рано усложнить систему. Сначала нужно подтвердить базовые границы сервисов, а уже потом расширять набор интеграций.
