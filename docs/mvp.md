# MVP

## Цель

Первая версия DJAI Platform должна проверить базовый сквозной сценарий платформы: пользователь открывает UI, отправляет сообщение, backend обращается к одному настроенному LLM endpoint и возвращает ответ. MVP нужен для проверки формы продукта, а не для решения всех задач сразу.

## Для кого

- внутреннее корпоративное использование
- администраторы
- технические пользователи
- ранние внутренние пользователи

## Что должно уметь MVP

- простой web-интерфейс на frontend
- отправка сообщения из UI в backend
- обработка запроса в backend
- подключение к одному настроенному LLM endpoint
- возврат ответа model в UI
- базовая конфигурация через переменные окружения
- простой локальный или on-prem сценарий запуска
- базовое логирование

## Что не входит в MVP

- agents
- сложный RAG
- multi-tenancy
- marketplace плагинов
- продвинутая admin-панель
- billing
- workflow engine
- enterprise SSO, AD и LDAP в первой версии
- orchestration между несколькими model

## Критерии готовности

- проект можно запустить локально или в простом on-prem окружении
- пользователь может открыть UI и отправить сообщение
- backend отправляет запрос в настроенный model endpoint
- ответ возвращается и показывается в UI
- базовая конфигурация описана в документации
- есть простая обработка ошибок и логирование основных событий

## Следующий шаг после MVP

После MVP логично усилить auth, добавить поддержку дополнительных model provider'ов, улучшить observability, начать knowledge/RAG слой и постепенно расширять административные возможности.
