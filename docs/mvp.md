# MVP

## Цель

Первая версия DJAI Platform должна проверить базовый сквозной сценарий платформы: пользователь открывает UI, отправляет сообщение, backend обращается к одному настроенному LLM endpoint и возвращает ответ. MVP нужен для проверки формы продукта, а не для решения всех задач сразу.

## Текущий статус

Этот узкий сценарий уже частично собран в репозитории:

- есть FastAPI backend и React/Vite frontend
- есть обычный и streaming chat path
- есть multi-turn история в памяти страницы
- есть runtime diagnostics, runtime-параметры и server-side limits
- есть базовая нормализация ошибок

Это ещё не означает, что платформа “готова”. За пределами этого chat slice по-прежнему ничего существенного не реализовано.

## Для кого

- внутреннее корпоративное использование
- администраторы
- технические пользователи
- ранние внутренние пользователи

## Что должно уметь MVP

- простой web-интерфейс на frontend
- отправка сообщения из UI в backend
- multi-turn диалог в рамках текущей страницы
- обработка запроса в backend
- подключение к одному настроенному OpenAI-compatible LLM endpoint
- возврат ответа model в UI
- streaming-ответ как основной путь
- базовая конфигурация через переменные окружения
- простой локальный или on-prem сценарий запуска
- базовое логирование и runtime diagnostics

## Что не входит в MVP

- agents
- сложный RAG
- multi-tenancy
- marketplace плагинов
- продвинутая admin-панель
- billing
- workflow engine
- enterprise SSO, AD и LDAP в первой версии
- orchestration между несколькими model
- database и persistence истории

## Критерии готовности

- проект можно запустить локально или в простом on-prem окружении
- пользователь может открыть UI и отправить сообщение
- backend отправляет запрос в настроенный model endpoint
- ответ возвращается и показывается в UI
- базовая конфигурация описана в документации
- есть простая обработка ошибок и логирование основных событий
- backend ограничивает слишком длинные сообщения и предсказуемо обрезает старую историю

## Следующий шаг после MVP

После этого узкого MVP логично сначала усилить auth/integration boundary и привести в порядок базовый platform skeleton, а уже потом двигаться к дополнительным model provider'ам, observability, knowledge/RAG и более широким admin-возможностям.
